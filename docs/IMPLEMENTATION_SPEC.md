# Постановка: веб-сервис транскрибации и саммари встреч

## 1. Цель

Создать веб-приложение, в котором пользователь загружает запись встречи и получает:

- транскрипт (`.txt`);
- транскрипт с метаданными (`.json`);
- транскрипт с разделением по спикерам (в режиме WhisperX);
- структурированное саммари на русском языке через локальный LLM (LM Studio, `qwen3-30b-a3b`) с retry policy по умолчанию `3`.

## 2. Границы MVP

- Поддерживаемые форматы: `.webm`, `.mp4`, `.m4a`, `.wav`.
- Язык транскрибации на MVP: русский (фиксированный).
- Режимы обработки:
  - `Whisper Large` (без спикеров);
  - `WhisperX` (с диаризацией по спикерам).
- Саммари включается опционально.
- Обработка должна быть асинхронной (очередь задач).
- У пользователя есть история задач и возможность скачать артефакты.

## 3. Пользовательские сценарии

1. Пользователь загружает файл, выбирает опции «со спикерами» и/или «сделать саммари».
2. Система создает задачу и показывает статус: `queued -> processing -> completed/failed`.
3. После завершения пользователь видит результаты в UI и может скачать файлы.
4. Пользователь может открыть список прошлых задач и повторно просмотреть результат.

## 4. Функциональные требования

- Загрузка файла с валидацией формата и размера.
- Создание задачи и постановка в очередь.
- Фоновая обработка без блокировки HTTP-запросов.
- Этапы пайплайна:
  - подготовка аудио/видео (`ffmpeg`);
  - транскрибация (`transcribe_webm.py` или `transcribe_whisperx.py`);
  - генерация саммари через LM Studio OpenAI-compatible API.
- Retry для саммари:
  - значение по умолчанию: `3`;
  - параметр должен быть конфигурируемым.
- Хранение артефактов и технических логов выполнения.
- API для статуса, просмотра и скачивания результатов.
- Понятная визуализация прогресса и ошибок в интерфейсе.

## 5. Нефункциональные требования

- Надежность: задачи не теряются при перезапуске backend.
- Производительность: тяжелые вычисления выполняются только в воркере.
- Наблюдаемость: логирование этапов, длительность этапов, причины ошибок.
- Безопасность:
  - проверка MIME-типа и размера файла;
  - изоляция файлов и результатов по пользователям;
  - защита секретов (`HF_TOKEN`, API ключи).
- Конфигурируемость через окружение:
  - `HF_TOKEN`;
  - `OPENAI_BASE_URL`;
  - `OPENAI_API_KEY`;
  - `SUMMARY_MODEL` (по умолчанию `qwen3-30b-a3b`);
  - `SUMMARY_RETRIES` (по умолчанию `3`);
  - `SUMMARY_RETRY_DELAY`.

## 6. Архитектура (предложение)

- Frontend: `React + TypeScript` (рекомендуется `Vite` + `React Router`).
- Backend API: `FastAPI`.
- Очередь задач: `Redis + Celery` (или `RQ`).
- Worker: отдельный Python-процесс для инференса.
- База данных: `PostgreSQL` (задачи, статусы, метаданные).
- Файловое хранилище:
  - MVP: локальная файловая система (`storage/`);
  - продакшен: S3/MinIO.
- LM Studio развернут на хосте с воркером и доступен по локальному endpoint.

## 7. API (черновой контракт)

### `POST /api/v1/jobs`

- Вход: файл + параметры (`mode=whisper|whisperx`, `summarize`, `min_speakers`, `max_speakers`, `summary_base_url`, `summary_model`, `summary_api_key`).
- Выход: `job_id`, `status=queued`.

### `GET /api/v1/jobs/{job_id}`

- Выход: статус, текущий этап, прогресс, сообщение об ошибке (если есть).

### `GET /api/v1/jobs/{job_id}/result`

- Выход: данные/ссылки на `txt`, `json`, `summary`.

### `GET /api/v1/jobs`

- Выход: список задач пользователя с пагинацией.

### `DELETE /api/v1/jobs/{job_id}`

- Удаление задачи и связанных артефактов (с учетом политики хранения).

## 8. Требования к frontend

- Страница загрузки:
  - drag-and-drop;
  - переключатели «со спикерами» и «сделать саммари»;
  - поле URL OpenAI-compatible endpoint для саммари;
  - поле API key для OpenAI-compatible endpoint;
  - поле модели саммари (default: `qwen3-30b-a3b`).
- Страница списка задач:
  - таблица/карточки со статусами и временем создания.
- Страница задачи:
  - прогресс этапов;
  - просмотр транскрипта;
  - просмотр саммари;
  - кнопки скачивания артефактов.

## 9. План реализации

1. Backend-скелет, схема БД, модель `Job`, базовые REST endpoints.
2. Очередь и воркер, интеграция текущих скриптов как библиотечных вызовов.
3. Хранение файлов и выдача результатов, обработка ошибок и ретраев.
4. Frontend (upload, list, job details, result viewer/download).
5. Логирование, метрики, документация и контейнеризация.

## 10. Критерии приемки MVP

- Через UI можно загрузить запись и получить транскрипт.
- В режиме WhisperX результат содержит разделение по спикерам.
- При включенном саммари формируется `.summary.md` через LM Studio.
- Retry для саммари с дефолтом `3` реально применяется.
- В UI корректно отображаются статусы `queued/processing/completed/failed`.
- После перезапуска backend задачи не теряются.
- Проект содержит документацию по локальному запуску всей системы.
